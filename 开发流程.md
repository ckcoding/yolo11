# AI 视觉监控中心 - 架构设计与开发说明

本文档描述当前项目的整体架构思路、核心设计取舍、接口职责以及后续扩展边界。目标不是写宣传材料，而是让后续开发、部署、排障时有一份可执行的技术说明。

---

## 一、目标与适用场景

本项目面向“实时视频流目标检测 + 前端实时展示 + 报警抓拍归档”的场景，当前重点是：

- 支持多路视频流并发检测。
- 支持图片单次识别。
- 支持前端低延迟查看检测结果。
- 支持报警抓拍事件回传与归档。
- 支持在不重启流任务的情况下动态调整检测目标。

当前方案优先服务以下场景：

- 局域网或受控网络中的监控值守页面。
- 桌面浏览器为主的实时观看场景。
- GPU 资源有限但需要多路并发的场景。

不建议直接把当前实现理解为“任意网络环境、任意终端、任意模型规模下的终局架构”。后续如果面向公网、多终端、更多模型种类，仍需要继续演进。



---

## 二、技术栈与职责划分

### 1. 后端服务

- `FastAPI`：提供任务控制、图片识别、状态查询和 WebSocket 事件推送。
- `Uvicorn`：承载 HTTP / WebSocket 服务。
- `OpenCV`：负责视频读取、图像编解码、绘制检测框。
- `Ultralytics YOLO`：负责目标检测推理。

### 2. 流媒体链路

- `FFmpeg`：把处理后的原始帧编码并推送到流媒体服务。
- `ZLMediaKit`：负责 RTMP 接入、HTTP-FLV/HLS/RTSP/WebRTC 等协议分发。

### 3. 前端

- `Vue 2` + `Element UI`：承载控制台页面。
- `mpegts.js`：在桌面浏览器中播放 HTTP-FLV。
- 原生 `WebSocket`：接收二进制 JPEG 预览帧和报警事件。

### 4. 结果存储

结果图和报警抓拍采用“可插拔存储”策略：

- 默认支持 `MinIO` 作为对象存储。
- `MinIO` 可以关闭。
- 关闭或上传失败时自动回退到本地静态目录 `artifacts/`。

这一点很重要：对象存储是增强项，不应该成为本地开发和单机部署的硬依赖。

## 三、核心架构设计

### 1. 共享模型池，而不是“每路流各自加载模型”

项目采用共享模型池，而不是为每一路视频流单独创建完整模型实例。

核心目的：

- 降低显存重复占用。
- 缩短新任务启动时间。
- 在多路并发时提高 GPU 利用率。

当前实现思路：

- 服务启动时，按 GPU 维度加载一组共享模型。
- 每张 GPU 对应一个批量推理服务。
- 每个流任务拿到的不是“独占模型”，而是一个“流控令牌 + 共享推理服务引用”。

这个思路对当前模型数量较少的场景是合适的。但要明确边界：

- 如果后续模型数量显著增加，启动时间和显存压力会随“模型数 x GPU 数”增长。
- 因此当前方案适合“少量固定模型”的阶段，不等于未来永远都应该全量预加载。

后续若模型规模继续扩大，建议演进为：

- 按模型组拆分服务。
- 按需装载冷门模型。
- 为不同任务类型设计不同的推理池。

### 2. 批量推理 + 隔帧推理

当前多路视频流并发的主要性能优化不只是“模型共享”，还包括：

- 批量推理：把多路流的帧汇聚后批量送入同一张 GPU。
- 隔帧推理：不是每一帧都送入推理，中间帧复用上一轮结果框。

这两点结合起来，能显著降低单位视频流的平均算力成本。

需要强调：

- 隔帧推理节省的是推理计算成本，不是零成本。
- 结果复用会带来轻微的时序滞后，属于典型的吞吐与实时性的工程折中。

### 3. 流媒体分发策略

当前项目不是单一路径播放，而是分成两种模式：

#### 模式 A：FLV 低延迟主链路

适用场景：

- 桌面浏览器实时观看。
- 需要较完整帧率和较稳定画面。

实现方式：

- 后端处理后的帧通过 FFmpeg 推到 ZLMediaKit。
- 前端通过 `mpegts.js` 播放 HTTP-FLV。

优点：

- 桌面 Chrome 类浏览器兼容性较好。
- 延迟较低。
- 适合主画面持续观看。

限制：

- 不适合作为所有终端的唯一方案。
- 某些浏览器和移动端兼容性不如 HLS。

#### 模式 B：WebSocket 预览微流

适用场景：

- 弱网环境下的预览。
- 不希望依赖完整流媒体播放链路。
- 希望快速直观看到检测结果。

实现方式：

- 后端按节流频率发送二进制 JPEG 预览帧。
- 前端直接用 `<img>` 刷新展示。

注意：

- 这里传的是二进制图片帧，不是 Base64 文本。
- 它的定位是“轻量预览 / fallback”，不是替代标准视频流的万能方案。

#### 关于 HLS / WebRTC

当前服务会返回相关播放地址，但应把它们理解为“可选分发能力”，不是当前方案的唯一主路径。

更准确的结论是：

- FLV：当前桌面低延迟主链路。
- WebSocket：当前轻量预览与弱依赖 fallback。
- HLS / WebRTC：可按部署环境和终端需求启用，不应在文档里被绝对化地判定为“彻底放弃”或“天然不可用”。

### 4. 动态调整检测目标

前端可以在任务运行期间动态修改检测目标，后端无需停止视频流。

这项设计的价值在于：

- 减少不必要的推理计算。
- 提高当前任务的算力利用效率。
- 降低页面操作成本。

这里要准确表述收益：

- 取消某个类目后，可以少做对应模型的推理工作。
- 但共享模型仍然可能驻留在显存中。
- 所以节省的主要是计算量与吞吐压力，不应写成“显存立即消失”。

### 5. 报警事件与抓拍归档

当检测到目标后，系统会：

1. 在主处理链路中完成检测与绘制。
2. 通过防抖逻辑控制重复报警频率。
3. 将抓拍图异步上传到对象存储或保存到本地目录。
4. 通过 WebSocket 向前端推送事件消息。

这里的关键设计点是：

- 抓拍存储放到异步线程池，避免阻塞视频主链路。
- 存储失败不影响主视频任务继续运行。
- 前端只保留有限数量的快照，避免长时间挂机导致页面内存持续增长。

## 四、当前后端处理流程

### 1. 视频流识别流程

1. 客户端调用 `/detect-stream` 提交视频源与检测目标。
2. 后端从模型池获取一个流控令牌。
3. 启动后台线程读取视频流。
4. 视频帧按设定频率进入批量推理服务。
5. 推理结果绘制到画面上。
6. 根据模式不同：
   - `flv`：推给 ZLMediaKit，由前端播放 HTTP-FLV。
   - `websocket`：压缩为 JPEG 预览帧，通过 WebSocket 发送。
7. 如果触发报警，则异步保存抓拍并向前端推送事件。
8. 任务结束后释放视频句柄、FFmpeg 进程、推理线程和模型池令牌。

### 2. 图片识别流程

1. 客户端调用 `/detect-image` 提交图片地址或本地路径。
2. 后端读取图片并解码。
3. 从模型池获取一个共享推理服务令牌。
4. 完成单次检测和结果绘制。
5. 保存原图与结果图。
6. 返回结果图地址和按类别统计结果。

### 3. 模型热更新流程

1. 停止现有流任务。
2. 等待运行中的处理线程退出。
3. 清空旧推理池。
4. 重新加载模型并重建共享池。

这类操作适合显式维护动作，不建议频繁触发。

## 五、接口职责总览

| 接口 | 方法 | 说明 |
|---|---|---|
| `/detect-stream` | `POST` | 创建视频流识别任务，返回任务 ID 与播放地址 |
| `/detect-image` | `POST` | 执行单张图片识别，返回结果图与统计信息 |
| `/update-classes` | `POST` | 动态更新某个流任务的检测目标 |
| `/ai/stop` | `POST` | 停止指定任务、别名下任务或全部任务 |
| `/tasks` | `POST` | 查询任务列表 |
| `/task-status` | `POST` | 查询单个任务状态 |
| `/ai/reload-models` | `POST` | 停止现有任务并重建模型池 |
| `/ws/{task_id}` | `WebSocket` | 预览帧和报警事件通道 |

说明：

- `ws/{task_id}` 不是普通 `GET` 接口，而是 WebSocket 端点。
- 前端和第三方接入方在使用时，应区分 HTTP 接口与 WebSocket 长连接。

## 六、部署与配置原则

### 1. 本地与容器双环境兼容

项目路径解析需要兼顾两种运行方式：

- Docker 容器内路径，例如 `/workspace/models/...`
- 本地开发目录路径，例如当前项目根目录下的 `models/`

因此配置不应强绑定某一种目录结构，而应允许合理回退。

### 2. 流媒体服务解耦

ZLMediaKit 是当前流媒体主链路的重要组件，但仍然属于独立服务，后端识别与流媒体分发应尽量解耦：

- 后端负责检测和结果帧生产。
- ZLM 负责流协议分发。
- WebSocket 微流可以在 ZLM 异常时作为有限可用的预览 fallback。

### 3. 对象存储可选

生产环境建议启用对象存储，原因是：

- 结果图便于跨实例访问。
- 更适合长期归档。

但本地开发和轻量部署允许关闭对象存储，直接使用本地目录，不应因为 MinIO 不可用而导致核心检测能力不可运行。

## 七、当前方案的适用边界

以下情况说明当前架构是合适的：

- 模型数量有限。
- GPU 数量固定且可控。
- 观看终端以桌面浏览器为主。
- 需要较低延迟而不是最强兼容性。

以下情况说明后续需要继续演进：

- 模型规模持续增长。
- 需要跨区域或公网大规模访问。
- 需要兼容移动端和 Safari。
- 需要更细粒度的多租户资源隔离。
- 需要长期历史事件检索、审核、回放与权限控制。

---

## 八、后续演进建议

建议按优先级逐步推进，而不是一次性重构：

### P1：稳定性

- 增加任务级监控指标，如帧率、推理耗时、队列积压、失败率。
- 增加流媒体异常与存储异常的统一日志字段。
- 为模型池、图片识别、停止任务接口补充自动化测试。

### P2：兼容性

- 根据终端类型自动选择播放协议。
- 为 Safari / 移动端补充 HLS 兜底策略。
- 为 WebSocket 微流增加更明确的码率和帧率配置。

### P3：扩展性

- 按模型类型拆分推理服务。
- 引入任务优先级与资源配额机制。
- 将事件记录从“仅推送”扩展为“可检索、可审计”的持久化记录。

---

## 九、结论

这套方案的主方向是合理的：

- 用共享模型池解决多路并发下的重复装载问题。
- 用批量推理和隔帧推理平衡吞吐与延迟。
- 用 FLV + WebSocket 组合解决当前实时展示问题。
- 用异步抓拍归档避免阻塞主视频链路。

但文档层面必须始终区分三件事：

- 当前实现是什么。
- 当前实现适合什么场景。
- 哪些只是当前阶段的工程取舍，而不是长期不变的终局架构。

只有这样，这份文档才能真正服务后续开发，而不是在后面反过来限制实现。
